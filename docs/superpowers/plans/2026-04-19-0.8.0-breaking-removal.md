# 0.8.0 — Breaking Removal of Deprecated Commands

> **For agentic workers:** Use superpowers:subagent-driven-development to execute task-by-task.

**Goal:** Deliver the breaking removal half of the 0.7.0 deprecation cycle. Remove the 17 commands that have been warning "removed in 0.8.0" since PR #51. Keep everything else (apply/plan/status/run/doctor/debug/query/explain/import/report) unchanged.

**Architecture:** Pure deletion + help reorganization. No new features. No refactor of the apply/planner/executor internals (those belong to 0.9.0). This release should be mechanically straightforward but requires careful test cleanup because many tests reference the deprecated shims.

**Tech Stack:** Node.js 24+, TypeScript, Commander, Vitest.

---

## Scope & non-goals

### In scope
- Remove all 17 commands deprecated in 0.7.0 (see matrix below).
- Delete `src/cli/deprecation.ts` entirely if no remaining callers.
- Remove `tests/cli/deprecation-*.test.ts` (legacy coverage for removed commands).
- Rewrite `flaker --help` to drop the "Deprecated" section.
- Delete now-orphaned command modules (`setup/init.ts` if merged into top-level, `exec/*` if merged, etc.).
- Keep legacy command *action functions* if they are shared by canonical commands. Only remove the registrations and no-longer-referenced action code.
- Migration reminder in CHANGELOG: users should have migrated during 0.7.x window via `docs/migration-0.6-to-0.7.md`.
- Re-run Iter 6 scenarios + Iter 6.5 hold-out as post-removal regression check.

### Non-goals
- `flaker apply --output <artifact>` / `--emit` → separate plan, 0.9.0
- Executor DAG / per-node status → 0.9.0
- `DesiredState` / `StateDiff` resource-diff model → 0.9.0
- `--target` / `--plan-file` / `--refresh-only` → 0.9.0
- Deleting `dev *` subtree (still kept as maintainer tools)
- Rewriting skills (they already point at canonical forms)

## Deprecated commands to remove (17)

Registration sites + what they alias:

| Registration | Removes this command | Canonical that replaces it |
|---|---|---|
| `src/cli/categories/setup.ts`     | `setup init` | `init` (top-level) |
| `src/cli/categories/exec.ts`      | `exec run`, `exec affected` | `run`, `run --gate iteration --changed ...` |
| `src/cli/categories/ops.ts`       | (none — `ops daily` was un-deprecated in 0.7.0 Phase 2) | — |
| `src/cli/categories/collect.ts`   | `collect ci`, `collect local`, `collect coverage`, `collect commit-changes`, `collect calibrate`, and the bare `collect` | `apply` |
| `src/cli/categories/quarantine.ts`| `quarantine suggest`, `quarantine apply` | `apply` |
| `src/cli/categories/policy.ts`    | `policy quarantine`, `policy check`, `policy report` | `apply` |
| `src/cli/categories/gate.ts`      | `gate review`, `gate history`, `gate explain` | `status --gate <name> [--detail] [--json]` |
| `src/cli/categories/analyze.ts`   | `analyze kpi`, `analyze eval`, `analyze flaky`, `analyze flaky-tag`, `analyze reason`, `analyze insights`, `analyze cluster`, `analyze bundle`, `analyze context`, `analyze query` | `status`, `status --markdown`, `status --list flaky`, `apply`, `explain <topic>`, `query <sql>` |
| `src/cli/categories/import.ts`    | `import report <file>`, `import parquet <dir>` | `import <file>` (auto-detect) |
| `src/cli/categories/report.ts`    | `report summary`, `report diff`, `report aggregate` | `report <file> --summary \| --diff \| --aggregate` |
| `src/cli/categories/debug.ts`     | `debug doctor` | `doctor` |
| `src/cli/main.ts`                 | top-level `kpi` alias | `analyze kpi` → now only `status` remains |

**Do NOT remove:**
- `ops daily` / `ops weekly` / `ops incident` (Advanced, kept)
- `debug retry` / `debug confirm` / `debug bisect` / `debug diagnose` (Primary under `debug`)
- `dev *` (hidden maintainer tools)
- `gate` category entirely — it has no canonical replacement for programmatic access beyond `status --gate`. The 3 subcommands are removed, but if the category becomes empty, decide per-case (drop the parent or leave an empty group with help text pointing to `status --gate`). **Recommend: drop.**

## Self-review

### Spec coverage
All 17 deprecated commands from the 0.7.0 deprecation matrix have an explicit removal task (Tasks 1-3). Help output is rewritten (Task 5). Version bump + CHANGELOG is captured (Task 6). Regression check is Task 7.

### Placeholders
None. Each task names the files to modify, the tests to delete or update, and the commit message.

### Type consistency
- Action-function exports kept: `analyzeKpiAction`, `analyzeReasonAction` (and the 4 other explain-backing actions), `setupInitAction`, `execRunAction`, `debugDoctorAction`, `runCollectCi`, `runCalibrate`, `runQuarantineSuggest`, `runQuarantineApply`. All are already referenced by the primary commands or by `apply`.
- If the subagent finds that a claimed "internal" action is not in fact referenced by anything after the shim goes, it should remove that action too in the same commit (report in status).

### Risk notes
- **Test cleanup is the biggest time-sink.** Roughly 20+ test files reference deprecated forms (anything with `analyze`, `collect ci`, `quarantine suggest` etc. in the file name or grep). Many will compile but fail at runtime because the spawned CLI no longer has those commands.
- **`apply` unaffected** — it invokes `runCollectCi` / `runCalibrate` / `runQuarantineSuggest` / `runQuarantineApply` as pure functions (not through the CLI). Task 2 must preserve those exports.
- **skills + docs** already point at canonical forms (from Phase 2). No skill/doc rewrite needed for 0.8.0.
- **`migration-0.6-to-0.7.md` stays** — remains a useful reference for users who defer migration until now. A second `migration-0.7-to-0.8.md` is not needed because the mapping is identical; 0.8.0 simply removes the shims the 0.7→0.8 transition only tightens.
